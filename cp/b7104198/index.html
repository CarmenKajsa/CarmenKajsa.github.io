初识调试

调试是什么？

调试就是代码在运行过程中，通过某种方式将他的运行状态暴露出来传递给开发者辅助进行问题排查和流程梳理。

常见的调试工具

Chrome DevTools

DevTools分为四部分，其中最主要的是 backend 和 frontend 两部分。

  • backend 调试器后端，负责把 Chrome 的网页运行时状态通过调试协议暴露出来。
  • frontend 调试器前端，负责对接调试协议，做 UI 的展示和交互。

这里涉及到的调试协议称为Protocol，也作CDP；而传输协议数据的信道是Message Channels。

VSCode Debugger

这个和Chrome DevTools相比多了一层适配器协议Debug Adapter Protocol。因为VSCode并不是纯JS的编辑器，因此调试其他语言就需要借助这个适配器协议进行识别。

Vue/React DevTools

Vue DevTools 或者 React DevTools 都是以 Chrome 插件的形式存在的，因此这里我们需要先搞清楚 Chorme 插件的机制

Chorme 插件

Chorme插件一般有两个部分组成：

  • Content Script：由页面的启动而生效，可以操作DOM并监听DOM Event
  • Background：从浏览器启动开始生效，生命周期较长，可以和 Content Script 还有 DevTools Page 通信。

如果是拓展 DevTools ,还会有一部分DevTools Page是在DevTools中显示，可以向当前 window 注入 JS 执行。

Vue DevTools

以Vue DevTools举例，他也分为 backend 和 frontend。

  1. backend 拿到 Vue 组件的信息，通过 window message 的方式传递给 Background。
  2. Background 和 DevTools Page 通信，实现消息转发。
  3. DevTools Page 根据拿到的数据，渲染组件的信息，实现交互功能。

React DevTools

React DevTools也是类似的，不过相比之下他多了一个自定义调试工具 Electron 。

总结

回顾几个常用的调试工具，不难发现他们都含有frontend、backend、调试协议、信道四个要素。其中：

  • backend：从前端获取运行时信息
  • frontend：渲染和交互
  • 调试协议：规定不同数据的格式
  • 信道

为什么 Chrome DevTools 和 VSCode Debugger 都可以调试网页？

因为他们俩的调试协议都是 CDP ，只不过 VSCode Debugger 多了一层 Debug Adapter Protocol 的转换。

launch/attach

在上面，创建 Chrome Debug 配置有launch 和 attach 两种方式，它们只是 request 的配置不同。

  • launch：启动网页，指定调试端口，然后 frontend 自动 attach 到这个端口。
  • attach：已经有一个跑起来的浏览器，那直接连接上就行。

使用attach

在命令行输入：/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --remote-debugging-port=9222 --user-data-dir=你自己创建的某个目录 ，这行代码的意思是手动把 Chrome 跑起来，指定调试端口 remote-debugging-port 为 9222，指定用户数据保存目录 user-data-dir 为你自己创建一个目录。
